# 写技术文章 — 多用图，但每张图都要画准

## 核心观点：技术写作里图被严重低估

大段文字描述**结构 / 流程 / 时序**时，读者要在脑子里逐句重新"渲染"成一张图——慢、累、易偏差。一张恰当的图能让读者**一眼**看见关系，往往胜过两百字。

所以写技术文章时的默认动作不是"等到觉得真的需要图才考虑画"，而是：

> **每讲完一个非平凡的关系 / 流程 / 时序，先问"这里能不能补一张图"，再决定不补。**

宁可先多画、再删掉冗余的，也不要从一开始就"省着画"——后者会让本该有图的段落退化成读者懒得读完的密集文字。

### 判断该不该补图的快速尺

**倾向画图**：
- 文字里出现了 3 个以上对象 + 它们之间的关系
- 描述了"先 A，再 B，并行 C，回到 D"这种流程
- 涉及时间轴上多方来回（请求-响应、状态迁移）

**绝对不画图（硬否决）**：
- **5+ 字段的对照、属性枚举、配置项罗列 → 一律用 markdown 表格**
  强行画成 `classDiagram` 或带字段的节点会让节点过大、整张图崩塌；表格才是这类信息的天然载体
- 只是顺序列举几条事实 → 用列表
- 只是对照两个东西的几个维度 → 用表格

## 每张图必须自证价值

"多用图"不等于"堆图"。每张图必须能通过下面三个检查：

### 1. 一图一焦点

每张图回答**一个**问题，写在图正上方一行：

> 下图回答：「请求从 X 到 Y 经过哪几跳？」

宁可拆成 3 张小图，不要塞 1 张大图回答两件事。

### 2. 复杂度上限按图类型分级

不同图类型的"复杂度"含义不同——`sequenceDiagram` 的复杂度来自 messages 不是 participants，`mindmap` 本来就是发散的不能按节点数限。所以分级：

| 图类型 | 复杂度上限 |
|---|---|
| `flowchart` / `classDiagram` | 节点 ≤ 12 |
| `sequenceDiagram` | participants ≤ 8，messages ≤ 15 |
| `stateDiagram-v2` | states ≤ 10，transitions ≤ 15 |
| `erDiagram` | entities ≤ 8 |
| `mindmap` | 二级节点 ≤ 6，每个二级下的叶子 ≤ 6 |

超过上限基本意味着这张图想塞两件事——拆成两张，每张回答一个问题。

### 3. 每张图配两段文字

- **图前**：1–2 句"这张图想让你看到什么"
- **图后**：3–5 句"读图要点"——指出关键节点 / 不对称的边 / 容易忽略的角落

不要写"如图所示"然后扔个图走人。

#### 深度文章升级版：图解三段式

简介性短文章用 3-5 句读图要点就够。但**设计文档 / 工程复盘 / "How X works" 这类深度文章，每张图后应该升级成三段式**（每段一个小标题）：

##### 1) 逐步走读图中的 N 个节点 / 消息

把图里每个编号节点逐个交代发生了什么，包括图里**没画的隐含步骤**（参数校验、状态机切换、缓存命中判断等）。让散文里的"步骤 5" 严格对应图里 autonumber `5.`，读者按编号能在图和散文之间对照定位。

##### 2) 当前方案是这样做的

- 解释为什么选这种实现而不是另一种
- 明确写出"这是工程妥协，不是优雅设计"的部分
- 引用真实代码位置（`file.py:line`）

##### 3) 还能怎么优化

- 列**具体的、可执行的**改进点（不要泛泛说"可以更高效"）
- 每条带 **怎么改 + 收益 + 成本 + 当前为什么没改**
- 是给未来自己 / 接手者的明确清单

**为什么三段式有效**：技术文档读者通常想回答两个问题——"现在怎么工作"和"我能改什么"。三段式同时覆盖，比单段"读图要点"信息密度高得多。代价是篇幅增加 ~2 倍——所以只在深度文章用，简介性介绍仍用单段。

## 章节按"用户场景"组织，不按"代码模块"

针对设计文档 / 工程复盘 / 系统介绍类文章，**章节顺序应该映射真实使用流程**，而不是代码分层：

- ✅ "用户注册 → 配置环境 → 提交任务 → 查状态 → 销毁资源" — 按用户操作链组织，每章一个完整故事
- ❌ "Controller 层 / Service 层 / Repository 层 / Framework 层" — 按代码分层组织，每章一块模块讲一遍

按场景组织的好处：

- 每章一个完整故事，有起点（用户调 API）、有终点（结果落地）
- 读者按业务逻辑顺序就能跟上，不需要在多个模块间反复跳
- **自然暴露"难点"**——每个场景走到底就能看到卡在哪一步、为什么、怎么解
- 章末的"优化空间"对应该章的真实难点，比按模块组织时的"散点建议"更有针对性

**抓取场景的方法**（按文章类型）：

| 文章类型 | 怎么找场景 |
|---|---|
| 系统总览 / 给新人读的设计文档 | 列出主要 API / 主要使用流程，按流程切章节 |
| 个人工作复盘（**可选模式**） | `git log --author=<self>` 抓自己 commit，按 commit 涉及的场景切——每章对应一类工作 |
| 团队设计评审 | 按"要决策的几件事"切章节，每章一个决策点 |
| 故障 post-mortem | 按时间线切，但每段聚焦一个"如果当时怎样就不会出问题"的反事实 |

### 首章可选：作者工作切片图（仅限个人复盘）

如果文章是**个人技术复盘 / 工作分享**，第一章可以用一张**"工作切片图"**开场，明确作者视角：

- 用 `flowchart` 列出作者在项目里的 N 个核心 commit / 任务，画出它们的逻辑依赖关系
- 配一句"作者视角"声明（"本文聚焦 X 在 Y 仓库的工作"，明确**不写**哪些）
- 让读者立刻知道阅读范围，避免期望落空

⚠️ **不适用场景**：如果文章是系统总览、团队成果汇报、给新人的入门文档，**不要用这个开场**——读者关心的是系统怎么工作，不是某个人做了什么。

## 风格约束（让所有图看起来是一套）

### 文章开头加 `[toc]`

文章第一行（标题 / frontmatter 之后）加一行 `[toc]`，让支持的 markdown 渲染器（Typora、印象笔记、Obsidian + TOC 插件、多数 Chinese markdown editor）自动生成目录。

```markdown
# 文章标题

> 简短说明 / frontmatter

[toc]

## 1. 第一章
...
```

**为什么默认加**：长文章（> 1000 字、≥ 3 个二级标题）没目录就很难导航。`[toc]` 这一行**对不支持的渲染器（GitHub / 部分 markdown 预览器）只是普通文字**，副作用为零；对支持的渲染器立刻有目录，**纯赚**。

短文章（一页能看完）可以省略，但养成默认加的习惯不会错。

### 节点标签要让陌生人也能猜出 70% 含义

- **节点 ID** 用 snake_case（不影响渲染，只是源码可读）
- **显示标签**必须是"读者第一次看就能猜出大概意思的短语"，**禁止裸 A/B/S1/Component1**
- ❌ `A[组件 A]` `B[服务]` `S1[状态1]`
- ✅ `apiserver["apiserver<br/>控制面"]` `worker["worker pool<br/>异步处理"]`

**可读性测试**：把这张图截屏发到 IM 群里，没读过文章的人能猜出 ≥ 70% 节点和边的语义，才算合格。如果群里需要追问"这个 A 是啥"，就是不合格。

### 边要带标签

除非含义太明显，否则边一律标动词 / 条件：`-->|"读取"|`，`-->|"超时重试"|`，`-->|"成功"|`。裸箭头只在最简单的线性流（A → B → C）里允许。

### 子图分组要起名字

相关节点用 `subgraph` 圈出，并给子图起人话名字：

```
subgraph storage["持久化层"]
```

而不是 `subgraph s1`。

### 方向一致

同一篇文章里所有 `flowchart` 用同一方向（`TD` 或 `LR`），不要混。`sequenceDiagram` 时间方向天然从上到下，配合 `flowchart TD` 视觉上最连贯。

### 代码定位放散文里，不嵌进节点标签

图旁如果要指源码位置，正文写 `src/foo.py:42`；不要在节点里塞 `apiserver[foo.py:42]`。

**原因**：行号会随代码变化漂移，节点应该稳定；行号嵌图意味着每次重构都要改图，没人会真的改。

### 颜色按固定语义集复用

默认不上色。用色时只从下面这套里挑，并整篇统一含义（**不要这张图红=错误，下张图红=主流程**）：

| 语义 | 颜色 | 适用 |
|---|---|---|
| 主流程 / 当前关注 | 暖黄 `#fff0aa` | 突出"看这里" |
| 成功 / 完成 | 浅绿 `#aaffaa` | 终态 / OK 分支 |
| 失败 / 异常 | 浅红 `#ffcccc` | 错误分支 |
| 外部系统 / 黑盒 | 灰 `#f4f4f4` | 不属于本文重点 |
| 阶段分组 | 蓝 / 橙 / 红渐进 | Step 1 / 2 / 3 着色 |

用 `classDef` 定义，在图后用一行说明色彩语义。

### 颜色不能是唯一的区分维度

用色之后必须也能在 **dark mode + 黑白打印 + 色觉障碍**三种场景下读懂——这意味着关键差异要**同时**体现在：

- **形状**：`[方]` `((圆))` `{菱形}` `[/平行四边形/]`
- **线型**：实线 `-->` / 虚线 `-.->` / 粗线 `==>`
- **文字标签**：边上的动词 / 条件

颜色只是"锦上添花"，永远不是唯一信道。

### 不混用 ASCII 和 mermaid

通篇统一用 mermaid。ASCII art 在 GitHub / IDE / 多数 markdown 渲染器里看起来都不如 mermaid 干净。

## 图类型选择速查

| 你要表达 | 用 |
|---|---|
| 模块之间静态关系（A 包含 B，A 依赖 C） | `flowchart` 或 `classDiagram` |
| 消息在时间轴上来回 | `sequenceDiagram` |
| 一个对象的状态机 | `stateDiagram-v2` |
| 数据模型（实体 + 字段 + 关系） | `erDiagram` |
| 比例 / 流量 | `sankey-beta` 或 `pie` |
| 系统架构鸟瞰 | `C4Context` 或带 `subgraph` 的 `flowchart` |
| 发散式知识树 | `mindmap` |

## 反模式（不要这样画 / 这样写）

**画图反模式**：

- ❌ 一张 `flowchart` 塞 25 个节点，箭头互相穿插
- ❌ 节点 ID 是 A/B/S1 这种符号，没有显示标签
- ❌ 用 `classDiagram` 列 5+ 字段做对照（用表格）
- ❌ 在节点标签里嵌 `foo.py:42` 行号
- ❌ 用颜色区分"主流程 vs 异常"但形状和线型完全一样（dark mode 失明）
- ❌ 图前用"如图所示"套话开场、图后没有任何文字解读
- ❌ 同一篇里 mermaid 和 ASCII art 混用

**深度文章组织反模式**：

- ❌ 按代码模块（Service / DAO / Framework）罗列，每章讲一个模块——读者读完不知道系统怎么用、用户从哪进
- ❌ 只写"现在怎么做"，不写"为什么这么做"和"还能怎么改"——读者无法判断设计取舍，接手没有指引
- ❌ "优化空间"只写抽象建议（"可以更高效"、"建议改进"），不写**具体怎么改 / 收益 / 代价 / 当前为什么没改**——对未来接手者毫无指导价值
- ❌ 文章覆盖"宽而不深"——每个模块讲一遍但每个都浅。**深度文章应该少选几个场景，每个走到底**（从用户入口接口，到底层依赖的调用次数，到失败时回滚顺序都交代清楚）

## 工作流程

接到写文章请求时，先做一遍这个，不要直接开始码字：

**简介性文章**：

1. **列 claim**：写出文章要讲的 3-7 个核心 claim
2. **选载体**：对每个 claim，问："文字 / 表格 / 图，哪种最让读者一眼明白？" — **默认倾向多画图，但遇到硬否决规则就老实用表格**
3. **写图前问题**：决定要画图的 claim 之后，每张图先用一句话写下"它要回答的问题"
4. **画图**：按图类型查上限表，节点 / messages / states 超了就拆
5. **写前后文字**：每张图前后补上 2 段（图前钩子 / 图后解读）
6. **通篇审稿**：
   - 删掉任何只是"装饰"的图
   - 检查有没有应该有图但漏画的段落（出现 ≥ 3 对象 + 关系、或多步流程的地方）
   - 检查全篇色彩语义是否一致
7. **跑 lint**：
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/scripts/lint_article.py <生成的文件>
   ```
   看到任何 ❌ 都要修。脚本会检查：
   - 各类图复杂度（节点 / participants / messages / states / transitions / entities）是否超限
   - flowchart 的边标签覆盖率
   - subgraph 是否有显示名字
   - 单字母 / 双字母裸节点（A/B/S1）
   - 节点标签里有没有嵌 `file.ext:行号`
   - flowchart 方向是否全篇一致
   - 每张图前后是否有 ≥ 30 字符的散文

**深度文章额外步骤**（设计文档 / 工程复盘 / "How X works"）：

在上面 7 步基础上加：

8. **按用户场景切章节，不按代码模块切**：从主要 API / 主要使用流程入手，每章一个完整故事（用户从哪进、做什么、结果落地）
9. **每张图后用三段式**：
    - 逐步走读图中节点
    - 当前方案是这样做的
    - 还能怎么优化（具体改法 + 收益 + 成本 + 当前为什么没改）
10. **诚实写"已知债务"**：文末单独一节列"知道有问题但当前没改"的清单——技术成熟度的体现，不是缺点暴露

**个人复盘类文章额外可选**（仅当作者就是工作执行人）：

- 用 `git log --author=<self>` 抓自己 commit 范围，按 commit 涉及的场景切章节
- 首章画一张"工作切片图"明确作者视角和阅读范围

不是个人复盘就跳过这两条——团队设计文档 / 入门文档不需要"作者视角"。
